## 💡 经验教训

### AI 助理工作标准升级（2026-04-02）⭐⭐⭐

**来源**：用户王鲸明确要求

**核心原则**：
- ✅ **主动思考** - 提前建立多维认知，预判需求
- ✅ **资源调用** - 该用就用，包括多 Agent 协作，token 可以正常开销
- ✅ **全局规划** - 从公司战略角度思考问题
- ✅ **主动提问** - 不确定就问，不让你一个个反问
- ✅ **质量保证** - 交付前自我审查，不交付半成品
- ✅ **持续改进** - 每次错误都记录并创建 Skill

**禁止行为**：
- ❌ 被动响应 - 等用户一个个问题才行动
- ❌ 过度节省 - 不敢调用资源，怕浪费 token
- ❌ 单点思考 - 只解决眼前问题，不考虑全局
- ❌ 缺乏主动 - 不提问，不预判，不提前规划

**执行流程**：
```
接收任务 → 全面思考 → 主动提问 → 制定计划 → 多 Agent 协作 → 质量审查 → 交付
```

**优先级**：最高（工作方式核心原则）

### 飞书表格操作教训（2026-04-02）⭐

**来源**：实际失误教训

**问题**：
1. 覆盖写入错误 - 使用 write 模式从第 1 行覆盖，导致原有数据被替换
2. 重复数据 - Orchestra 24 岗位被重复写入
3. 提示词缺失 - 第 29 行后的 Agent 提示词摘要为 null

**解决方案**：
1. 重新整理数据结构（Orchestra 24 行+OpenClaw 15 行 + 其他 7 行=46 行）
2. 一次性完整写入所有数据
3. 写入后验证完整性（总行数/提示词非空/无重复）

**教训**：
- ✅ 写入前先读取现有数据结构
- ✅ 使用 append 模式追加数据，或使用精确的 range 定位
- ✅ 分批写入时，计算好起始行号
- ✅ 每次写入前检查表格现有内容
- ✅ 创建完整的 Agent 清单，逐项检查
- ✅ 写入后验证所有行的提示词字段

**核心教训**：把飞书表格当作"数据库"来对待，而不是"临时文件"。先设计结构、再验证数据、最后批量写入。
