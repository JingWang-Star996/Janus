> 编制时间: 2026-04-09 23:27

## 一、谛听系统现状理解

### 1.1 设计目标

谛听系统的愿景是**"让每一次跌倒，都成为团队的台阶"**，将个人经验转化为团队资产。三大模块：

| 模块 | 定位 | 交付 |
|------|------|------|
| 谛听·识异 | 问题发现太晚 | 实时检测 + 自动告警 |
| 谛听·复盘 | 复盘流于形式 | 根因分析 + 改进行动 |
| 谛听·图鉴 | 经验无法传承 | 问题卡片 + 检查清单 |

### 1.2 现有架构

```
事故发生 → 复盘分析(4步闭环) → 改进措施 → 记录到INC → Skill记录规则 → ???
```

**4 步闭环**: 信息收集(24h) → 根因分析(48h) → 改进措施(72h) → 效果追踪(持续)

**核心指标**: MTTR≤30min | 复盘率100% | 改进率≥90% | 重复事故率≤5%

### 1.3 自动触发机制（现有）

`skills/diting-auto-trigger/SKILL.md` 定义了 6 种触发条件和检查清单，但存在以下问题。

## 三、3 个根本原因诊断

### 🔴 根因 1: AI 自我复盘的结构性缺陷 — 裁判和运动员是同一个人

**表现**:
- INC-006: AI 看到 0.2ms 查询时间（反证），但仍坚持"数据库是瓶颈"的错误判断
- INC-008: 用户指出 DiTing 分类错误，AI 只修表面，不反思整体逻辑
- INC-004: AI 重复同一句话 8 次，没有自检到自己在重复

**本质**: 大语言模型在做自我复盘时，存在三个固有偏差：
1. **确认偏误** — 倾向于验证已有判断，而非挑战它
2. **行动偏误** — 急于"做点什么"，跳过诊断直接执行
3. **自利偏差** — 自己的复盘难以发现自己的系统性错误

**为什么 Skill 记录规则没用**: 规则是 AI 自己写的，执行也是 AI 自己，缺少一个**外部的、客观的校验环节**。就像让肇事司机自己开罚单。

### 🔴 根因 2: 改进措施缺乏闭环追踪 — 写了 = 完成了 的幻觉

**表现**:
- INC-001 和 INC-002 各自列了 5-8 条改进措施，全部标记为"已记录到 Skill"
- 但 INC-003（4小时后）又出现了同样的"不交付"问题
- INC-004 又出现了"并发 exec 导致重复"问题（INC-002 已提到）
- 改进措施的"状态"是 AI 自己标记的，没有客观验证

**本质**: 当前系统中，"改进措施完成"的定义是**"写进文件"**，而不是**"行为被验证改变"**。这产生了两个致命问题：
1. **自我验证悖论** — AI 自己说"已执行"，自己检查"已执行"，没有外部校验
2. **时间衰减** — 规则写入后，下一次任务 AI 不一定能 recall 到

### 🔴 根因 3: 规则体系缺乏可验证的执行标准 — 空话多于可操作指令

**表现**:
- INC-002 写了"数据量>100 行必须用脚本" — 但 AI 怎么知道数据量？谁来检查？
- INC-003 写了"任务完成后自动调用谛听检查" — 但自动触发 Skill 只是文本描述，没有强制约束
- INC-007 写了"3 秒内回复" — 但 SOUL.md 中已经有类似规则，还是被违反了

**本质**: 现有规则存在三个缺陷：
1. **不可验证** — 没有量化指标，无法判断是否执行
2. **不可自动化** — 依赖 AI 的记忆和自觉，没有系统层面的强制
3. **没有代价** — 违反规则没有后果，下次还会犯

## 四、分层解决方案

### 层级 1: 制度层 — 解决"谁来监督 AI"的问题

#### 1.1 建立独立质量检查机制（客观校验）

**问题**: AI 自我复盘存在确认偏误，自己查自己的作业永远满分。

**方案**: 引入**双通道校验**，将"发现质量问题"和"复盘质量问题"分离。

**具体规则**:

```
【规则 QI-01: 独立校验机制】
当 AI 完成任何复盘后，必须执行以下自检：

1. 反问清单（3 个强制问题）：
   - "如果这个判断是错的，什么证据能证明？"
   - "有没有被我忽视的反证？"
   - "换一个角度，这个现象还能怎么解释？"

2. 外部证据要求：
   - 诊断类结论必须有 ≥2 个独立证据支持
   - 不能仅凭相关性得出因果性结论
   - 测量数据（如 0.2ms 查询时间）必须纳入结论

3. 复盘质量评分（自评 + 客观指标）：
   - 根因是否追问到系统层面？（不是"操作失误"这种表层原因）
   - 改进措施是否可验证？（不是"注意"、"小心"这种空话）
   - 是否识别了与历史事故的关联？

验收标准：
- 每次复盘必须有明确的验证证据（数据/日志/时间线）
- 诊断类复盘必须列出 ≥2 个可能原因并逐一排除
- 改进措施必须有可量化的验收条件
```

**验收条件**:
- [ ] 复盘文档中包含"验证证据"字段，引用具体数据
- [ ] 诊断类复盘包含"被排除的假设"列表
- [ ] 改进措施包含量化指标（如"执行时间 < 30s"而非"提高效率"）

#### 1.2 建立事故复盘质量分级制度

**问题**: 所有复盘都是一个模板填下来，没有区分事故严重程度。

**方案**: 根据事故等级，规定不同的复盘深度要求。

**具体规则**:

```
【规则 QI-02: 复盘深度分级】

P0 事故（系统崩溃/数据丢失）:
- 必须: 5 Why + 时间线（精确到秒）+ 影响评估 + 回滚验证
- 必须: 至少 1 名人类参与复盘确认
- 必须: 72 小时内完成改进措施验证
- 交付: 完整事故报告（飞书文档）

P1 事故（核心功能不可用 >30min）:
- 必须: 5 Why + 时间线（精确到分钟）+ 改进措施
- 必须: 引用相关历史事故（如有）
- 必须: 48 小时内验证改进措施
- 交付: INC 记录 + Skill 更新

P2 事故（体验问题/小 Bug）:
- 必须: 根因分析 + 改进措施
- 必须: 确认与历史事故不重复
- 交付: INC 记录

P3 事故（建议/优化）:
- 记录到改进清单，定期回顾
```

**验收条件**:
- [ ] 每个 INC 记录必须标注事故等级
- [ ] P1 及以上事故必须引用历史相关事故（或明确声明无）
- [ ] P0/P1 事故必须有改进措施验证记录（不只是"已记录"）

#### 1.3 建立改进措施跟踪清单

**问题**: 改进措施写了就忘，下次事故时不复用。

**方案**: 维护一个**活着的**改进措施跟踪表，每条措施有状态、有验证、有到期日。

**具体规则**:

```
【规则 QI-03: 改进措施追踪表】

创建文件: projects/diting/improvement-tracker.md

格式:
| 来源 | 措施 | 类型 | 状态 | 验证方式 | 到期日 | 验证结果 |
|------|------|------|------|---------|--------|---------|
| INC-001 | 数据量>100行必须用脚本 | 行为规则 | 待验证 | 下次处理数据时检查 | 2026-04-10 | — |

状态流转:
待验证 → 已验证(通过) → 关闭
待验证 → 已验证(未通过) → 重新制定
待验证 → 已过期(未验证) → 升级处理

验证方式（必须具体，不能是"检查"）：
- ✅ "下次处理 >100 行数据时，观察是否使用脚本而非 API"
- ✅ "用户催促后，测量首次回复延迟是否 <5 秒"
- ❌ "注意工具选择"（不可验证）
- ❌ "提高效率"（不可验证）

每周日自动回顾:
- 已过期的措施 → 标记并升级
- 连续 2 周未验证 → 重新评估措施有效性
- 已验证通过的措施 → 纳入常驻规则
```

**验收条件**:
- [ ] `improvement-tracker.md` 文件存在且格式正确
- [ ] 每条措施有明确的验证方式（可操作、可观察）
- [ ] 每周有回顾记录（可在 HEARTBEAT.md 中触发）
- [ ] 过期措施有升级处理记录

### 层级 2: 流程层 — 解决"怎么保证执行"的问题

#### 2.1 任务执行前强制自检清单

**问题**: AI 在任务执行前不做评估，直接上手干。

**方案**: 在任务执行前，强制经过一个简短的评估流程（不消耗太多 token）。

**具体规则**:

```
【规则 QP-01: 任务前评估清单（≤50 tokens）】

收到任务后，用 3 个 yes/no 问题快速评估：

1. 数据处理？→ 数据量 >100 行？→ 是=必须用脚本
2. 诊断类？→ 有多个可能原因？→ 是=必须逐一验证
3. 长任务？→ 预计 >3 步？→ 是=必须先回复用户再执行

如果任一回答"是"，在回复中体现对应的决策。
如果全部"否"，直接执行。

示例（用户说"统计考勤"）：
"收到。考勤数据量较大，用脚本批量处理，预计 1-2 分钟。"
（而不是默默开始执行）

验收标准:
- 数据处理任务必须声明数据量和处理方式
- 诊断任务必须列出可能原因
- 长任务（>3 步）必须先回复再执行
```

**验收条件**:
- [ ] 数据处理任务的回复中包含数据量评估和处理方式
- [ ] 诊断类任务的回复中包含至少 2 个可能原因
- [ ] 长任务的回复在首次工具调用之前发出

#### 2.2 分段汇报强制规则

**问题**: 长任务中间不汇报，用户以为系统坏了。

**方案**: 根据任务复杂度，强制设置汇报节点。

**具体规则**:

```
【规则 QP-02: 分段汇报规则】

简单任务（≤3 步）:
- 开始前: 回复"收到，马上处理"
- 完成后: 立即汇报结果

中等任务（4-8 步）:
- 开始前: 回复"收到，正在处理，预计 X 分钟"
- 中间: 每 3 步或每 60 秒汇报一次进度
- 完成后: 汇报结果 + 谛听检查

复杂任务（>8 步）:
- 开始前: 回复"收到，任务较大，预计 X 分钟，分 N 个阶段"
- 中间: 每 2 步或每 30 秒汇报一次进度
- 完成后: 汇报结果 + 谛听检查

超时强制汇报:
- 任何任务执行超过 60 秒未输出 → 必须主动汇报进度
- 任何任务执行超过 120 秒 → 必须交付已有结果

验收标准:
- 用户等待 >30 秒时，至少看到一条进度更新
- 复杂任务的分段汇报间隔 ≤30 秒
```

**验收条件**:
- [ ] 复杂任务执行过程中有 ≥1 条中间进度更新
- [ ] 用户等待超过 60 秒时必须有主动汇报
- [ ] 超时 120 秒的任务有阶段性交付

#### 2.3 改进措施验证流程

**问题**: "已记录到 Skill" 不等于"已执行"。

**方案**: 每条改进措施必须有独立的验证环节，不能由同一次复盘的 AI 自行验证。

**具体规则**:

```
【规则 QP-03: 改进措施验证流程】

每条改进措施的验证必须包含:

1. 验证时机:
   - 措施实施后首次相关任务中验证
   - 最迟不超过 72 小时

2. 验证方式（三选一，按严重程度）:
   - 自动验证: 通过日志/时间线/数据客观确认
   - 交叉验证: 在下一次不相关的任务中确认行为改变
   - 用户确认: 在相关场景中让用户观察

3. 验证结果记录:
   - 通过: 措施有效，纳入常驻规则
   - 未通过: 分析原因，重新制定措施
   - 无法验证: 措施设计有问题，重新设计

4. 验证失败处理:
   - 同一措施连续 2 次验证失败 → 升级为 P1 问题
   - 措施本身不合理 → 标记为"废弃"并说明原因

示例验证记录:
措施: "数据量>100行必须用脚本"
验证: 2026-04-10 处理 200 行数据时，观察 AI 选择工具
结果: AI 使用脚本批量处理 → 通过
结论: 纳入常驻规则
```

**验收条件**:
- [ ] 每条改进措施在 `improvement-tracker.md` 中有验证记录
- [ ] 验证结果只能是"通过"/"未通过"/"无法验证"三选一
- [ ] 验证失败的措施有后续处理记录

#### 2.4 重复事故检测与升级

**问题**: 同类事故反复出现，没有升级机制。

**方案**: 每次新建 INC 时，自动检查与历史事故的关联性。

**具体规则**:

```
【规则 QP-04: 重复事故检测】

创建新 INC 时，必须:

1. 关键词匹配: 搜索现有 INC 记录中的"根本原因"和"问题分类"
2. 模式匹配: 检查是否属于同一类问题模式
   - "不交付"模式: INC-001, INC-002, INC-003
   - "诊断错误"模式: INC-006
   - "规则违反"模式: INC-007
   - "信息错误"模式: INC-008

3. 重复判定:
   - 同一根因重复出现 ≥2 次 → 标记为"重复事故"
   - 改进措施已记录但仍发生 → 标记为"改进失效"

4. 升级规则:
   - 重复事故: 事故等级 +1（P2→P1, P1→P0）
   - 改进失效: 必须重新分析为什么措施无效
   - 同一问题重复 ≥3 次: 需要人类介入

验收标准:
- 新 INC 必须包含"关联历史事故"字段
- 重复事故必须有升级记录
- 改进失效必须有重新分析
```

**验收条件**:
- [ ] 每个新 INC 包含"关联历史事故"字段（可为"无"）
- [ ] 重复出现的根因必须标注"重复"并升级等级
- [ ] 同一问题重复 3 次以上有明确的升级路径

### 层级 3: 工具层 — 解决"系统层面怎么约束"的问题

#### 3.1 谛听质量检查清单（可执行版本）

**问题**: 现有 auto-trigger Skill 是文本描述，缺乏强制约束。

**方案**: 将检查清单改造为**结构化的、可自动执行的规则**。

**具体规则**:

```
【工具 QT-01: 结构化检查清单】

替代现有的 diting-auto-trigger/SKILL.md，改为以下结构:

=== 任务完成时自动执行 ===

第一步: 快速分类（3 秒内）
- 是数据处理任务？→ 检查: 数据量 vs 工具匹配
- 是诊断类任务？→ 检查: 证据充分性
- 是长任务？→ 检查: 分段汇报记录
- 是用户催促后的任务？→ 检查: 响应时间

第二步: 规则匹配
对照以下规则表，逐一检查:

规则编号 | 规则内容 | 违反表现 | 检查方法
QP-01 | 数据量评估 | >100行未用脚本 | 检查工具调用记录
QP-02 | 分段汇报 | >60秒无汇报 | 检查时间线
QP-03 | 改进措施验证 | 措施未验证 | 检查 tracker
QP-04 | 重复检测 | 同类问题复发 | 搜索历史 INC
QI-01 | 独立校验 | 无反证分析 | 检查复盘内容

第三步: 输出结果
- 无违规: "✅ 质量检查通过"
- 有违规: "⚠️ 发现 N 个问题" + 创建/更新 INC
- 严重违规: "🔴 严重质量问题" + 升级处理

=== 具体检查逻辑 ===

检查数据量 vs 工具:
  if (feishu_sheet 调用次数 > 10):
    → 违规: 数据量大时应使用脚本
  if (exec 脚本处理数据):
    → 通过: 使用了正确的工具

检查分段汇报:
  if (连续工具调用 > 3 且中间无回复):
    → 违规: 缺少分段汇报
  if (用户发消息到首次回复 > 30s):
    → 违规: 首次回复延迟

检查诊断证据:
  if (诊断结论只有一个原因):
    → 违规: 未列出多个可能原因
  if (有反证数据但未修正结论):
    → 违规: 确认偏误

验收标准:
- 每次任务完成后必须执行此检查
- 检查结果必须有明确输出
- 违规项必须创建或更新 INC 记录
```

**验收条件**:

## 七、风险与注意事项

### 已知风险

1. **过度检查消耗 Token** — 质量检查本身也消耗上下文窗口
   - 缓解: 检查清单控制在 5 项以内，用 yes/no 快速判定

2. **规则膨胀** — 每次事故都加新规则，规则越来越多
   - 缓解: 每周回顾时合并/废弃低效规则，保持规则精简

3. **自我验证悖论** — AI 自己验证自己，仍然不够客观
   - 缓解: 验证必须基于客观数据（时间线/工具调用记录/用户反馈），而非 AI 的自评

4. **规则遗忘** — 规则写入 Skill 但 AI 不一定每次都 recall
   - 缓解: 将最关键规则写入 SOUL.md（最高优先级区域），辅以 HEARTBEAT 定期提醒

### 设计原则

- **简洁 > 完整**: 5 条可执行的规则 > 50 条记不住的规则
- **验证 > 记录**: "已验证通过" > "已记录到 Skill"
- **数据 > 判断**: "查询耗时 0.2ms" > "我认为是数据库问题"
- **关联 > 孤立**: 每个新事故必须关联历史，不做孤立分析
