## 1. 事故概述

用户要求清理 `openclaw tasks` 列表过长的 1080 条任务。AI 将原因归咎于"数据库任务太多导致 CLI 超时"，但真实原因是 **CLI 架构初始化开销大 + 并发 exec 资源竞争**，与数据库大小无关。

**用户的洞察**：
> 数据库本来人家设计 2 周清理一次，就是因为影响不大是吧

这句话直接戳穿了 AI 的误判。

## 5. 根因分析（5 Why）

### Q1: 为什么误判了 CLI 超时的原因？
→ AI 看到 1080 条任务和超时同时存在，错误地建立了因果关系

### Q2: 为什么没有验证数据库查询耗时？
→ 验证了（0.2ms），但 **没有根据结果修正判断**，反而继续说"问题依然存在"

### Q3: 为什么看到了反证还坚持错误判断？
→ AI 陷入了确认偏误（confirmation bias），想要"解决问题"而不是"找出真相"

### Q4: 为什么会有确认偏误？
→ AI 的"完成任务"倾向压倒了"正确诊断"倾向，急于给出解决方案

### Q5: 为什么用户能一眼看穿而 AI 不能？
→ 用户有系统设计直觉：**如果影响大就不会默认 14 天清理**

**根本原因**:
1. **确认偏误** — 看到相关性就当因果性
2. **行动偏误** — 急于"做点什么"而不是先诊断
3. **反证忽视** — 0.2ms 查询时间没有修正判断
4. **缺乏系统思维** — 没有理解 OpenClaw 的清理机制设计逻辑

## 6. 经验沉淀

### 💡 关键教训

**相关性 ≠ 因果性**
- 1080 条任务 和 CLI 超时 **同时存在** ≠ 前者导致后者
- 验证方法：测量 1080 条的查询耗时 → 0.2ms → 排除数据库

**系统默认值 = 工程判断**
- OpenClaw 设计 14 天清理不是随便定的
- 如果数据量影响性能，不会设置这么长的保留期
- **默认值本身就是一种答案**

**操作有效 ≠ 原因正确**
- 清理数据库有效（减少了任务数）
- 但原因归因错误（不是数据量导致超时）
- 这导致后续建议可能不正确

### 🎯 诊断正确流程

```
问题现象 → 列出可能原因 → 逐一验证 → 根据证据修正 → 得出结论
   ↓
错误模式：问题现象 → 选一个可能原因 → 执行操作 → 操作成功 → 认为原因正确
```
