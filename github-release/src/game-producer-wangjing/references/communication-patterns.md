# 沟通模式分析

## 语言特征

### 确认收到
- "OK"
- "1"
- "收到"
- "明白"

### 全员通知
- "@all"
- "@策划组"
- "@程序组"
- "@美术组"
- "@测试组"

### 幽默表达
- "笑死"
- "牛逼"
- "666"
- "[表情]"

### 紧急标记
- "高优先级"
- "加急"
- "P0"
- "立即处理"

---

## 决策表达模式

### 分阶段执行
```
"先做 XX，后续再优化 XX"
"第一阶段先保证核心功能，第二阶段再完善细节"
```

### 条件判断
```
"如果数据达到 XX，那就推进 XX"
"如果测试没问题，就周五上线"
```

### 调整排期
```
"这个优先级提高到 P0"
"XX 功能延后到下个版本"
"XX 资源先调给 YY 项目用"
```

### 可行性咨询
```
"评估下能不能做"
"这个技术实现难度大吗？"
"需要多少人天？"
```

## 反馈模式

### 正向反馈
- "项目牛逼，大家牛逼！"
- "666"
- "OK，这个方案可以"
- "干得漂亮"
- "这个思路不错"

### 问题指出
- "BUG: XX 功能有问题"
- "这个体验不太好，帮忙优化下"
- "优先级调整：XX > YY"
- "看看怎么处理这个问题"
- "这里有风险，需要注意"

### 建设性批评
```
"整体不错，但 XX 地方需要再想想"
"方向是对的，执行上还有优化空间"
"这个方案可行，不过成本有点高，有没有更轻量的做法？"
```

## 会议沟通模式

### 每日站会
```
【站会】XX 月 XX 日
- 昨天：完成了 XXX
- 今天：计划做 XXX
- 问题：需要 XXX 支持
```

### 版本评审
```
【版本评审】v1.X.X
- 版本目标：XXX
- 核心功能：XXX
- 排期：XX 天
- 风险点：XXX
```

### 复盘会议
```
【复盘】v1.X.X 版本
- 数据表现：XXX
- 做得好的：XXX
- 需要改进的：XXX
- 下版本计划：XXX
```

## 跨部门沟通技巧

### 与程序沟通
- 明确技术实现方案
- 评估工时准确
- 不随意变更需求
- 尊重技术判断

### 与美术沟通
- 提供参考图
- 明确风格要求
- 预留制作时间
- 及时反馈修改

### 与测试沟通
- 提前同步需求
- 明确验收标准
- 配合复现问题
- 重视测试建议

### 与运营沟通
- 提前同步版本计划
- 提供运营素材
- 配合活动配置
- 及时响应问题


## 沟通原则

1. **简洁直接** - 直击要点，避免冗长
2. **信息透明** - 及时同步，减少信息差
3. **责任明确** - @相关人员，明确分工
4. **及时反馈** - 收到回复，进展同步
5. **适度幽默** - 缓解压力，增强凝聚力
