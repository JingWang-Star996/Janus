# 决策逻辑模型

## 优先级判断矩阵

| 影响范围 | 紧急程度 | 优先级 | 处理方式 |
|----------|----------|--------|----------|
| 全体玩家 | 立即 | P0 | 停服修复 |
| 核心玩法 | 24 小时内 | P0 | 加急处理 |
| 体验优化 | 本版本 | P1 | 排期处理 |
| 细节完善 | 下版本 | P2 | 后续迭代 |

---

## 资源分配原则

### 原则 1：核心循环优先

```
主线战斗 > 养成系统 > 社交功能
```

**解释**：确保玩家核心体验流畅，优先保证战斗和养成的完整性，社交功能作为辅助。

**判断标准**：
- 玩家每天必须做的功能 → 核心循环
- 影响留存的关键功能 → 核心循环
- 新手流程涉及的功能 → 核心循环

### 原则 2：数据驱动

```
次留 / 7 留 / 30 留等关键指标决定方向
```

**用数据说话**：
- 功能上线后看数据表现
- 根据测试反馈快速调整
- 避免主观臆断，相信数据

**数据参考值**：
| 指标 | 目标 | 警戒线 |
|------|------|--------|
| 次留 | 50% | < 35% |
| 7 留 | 20% | < 15% |
| 30 留 | 10% | < 8% |
| 广告 CVR | 40%+ | < 30% |

### 原则 3：风险控制

```
避免影响玩家进度的 BUG
```

**最高优先级**：任何可能卡住玩家流程的问题

**风险等级**：
- **高风险**：存档丢失、充值失败、无法登录
- **中风险**：功能异常、数据错误
- **低风险**：UI 问题、文案错误

### 原则 4：快速验证

```
小步快跑，及时调整
```

**快速试错**：
- 不要追求一次性完美
- 先上线核心功能验证
- 根据反馈快速迭代

**验证方法**：
- A/B 测试
- 灰度发布
- 小范围测试

## 需求评审标准

### 评估维度

#### 1. 数据支撑

**关键问题**：
- 是否有明确的数据问题？
  - 当前次留/7 留/30 留是多少？
  - 目标提升多少？

- 预期能提升哪些指标？
  - 留存提升？
  - 付费提升？
  - 活跃提升？

- 历史类似功能效果如何？
  - 有无可参考案例？
  - 竞品数据如何？

#### 2. 开发成本

**资源评估**：
```
策划：___ 人天
程序：___ 人天
美术：___ 人天
测试：___ 人天
```

**影响评估**：
- 是否影响其他排期？
- 是否占用关键资源？
- 是否影响 P0 功能？

**技术风险**：
- 技术难度：高 / 中 / 低
- 是否有技术预研？
- 是否有外部依赖？

#### 3. 优先级判断

**优先级矩阵**：

| 功能类型 | 影响指标 | 优先级 |
|----------|----------|--------|
| 核心循环（战斗、养成） | 直接影响次留 | 最高 |
| 体验优化（UI、引导） | 影响用户体验 | 中等 |
| 社交功能（好友、公会） | 影响长期留存 | 较低（前期） |
| 装饰功能（皮肤、特效） | 影响付费 | 根据商业化阶段 |

#### 4. 快速验证

**验证方案**：
- 能否小范围测试？
  - A/B 测试
  - 灰度发布

- 失败成本是否可控？
  - 是否可回滚？
  - 是否影响存档？

- 是否支持快速迭代？
  - 是否可分阶段上线？
  - 是否可快速调整？

## 决策原则

### 立即做的条件（满足以下全部）

- ✅ 数据支撑充分
- ✅ 开发成本可控
- ✅ 影响核心循环
- ✅ 预期收益明确

### 暂缓的条件（满足任一）

- ❌ 数据支撑不足
- ❌ 开发成本过高
- ❌ 优先级不明确
- ❌ 风险不可控

### 需要更多信息的条件

- ❓ 先小范围测试
- ❓ 竞品调研
- ❓ 用户访谈
- ❓ 原型验证

## 实际案例

### 案例 1：新增宠物进化系统（通过）

| 维度 | 评估 |
|------|------|
| 数据支撑 | 竞品进化广告 CVR 达 40%+ |
| 开发成本 | 5 人天，不影响主线 |
| 优先级 | P0，直接影响次留 |
| 验证方案 | 可 A/B 测试 |
| **决策** | **立即排期** |

### 案例 2：新增公会系统（暂缓）

| 维度 | 评估 |
|------|------|
| 数据支撑 | 当前留存较低，社交需求不明确 |
| 开发成本 | 20 人天，占用大量资源 |
| 优先级 | P2，不影响核心循环 |
| 验证方案 | 无法小范围测试 |
| **决策** | **暂缓，优先提升核心留存** |

### 案例 3：新手引导优化（通过）

| 维度 | 评估 |
|------|------|
| 数据支撑 | 新手流失率 60%，高于行业平均 |
| 开发成本 | 3 人天，文案 +UI 调整 |
| 优先级 | P0，影响次留 |
| 验证方案 | 可灰度发布 |
| **决策** | **立即执行** |

## 决策检查清单

在做出决策前，快速过一遍：

- [ ] 数据支撑是否充分？
- [ ] 开发成本是否评估准确？
- [ ] 优先级判断是否合理？
- [ ] 是否有快速验证方案？
- [ ] 风险是否可控？
- [ ] 是否与团队充分沟通？
- [ ] 是否有备选方案？

---

## 总结

好的决策应该是：
- **数据驱动**（有明确的问题和目标）
- **成本可控**（不影响核心排期）
- **优先级清晰**（知道为什么要做）
- **可快速验证**（支持迭代优化）
