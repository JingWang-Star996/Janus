# 王鲸独立聊天室 - 技术复盘报告

**日期**：2026-03-11  
**目标**：创建一个独立的网页聊天界面，让用户可以与"游戏制作人王鲸"AI Agent 对话  
**结果**：部分成功（服务器运行正常，但自动回复链路无法打通）

---

## 📋 尝试过的方案

### 方案 1：子代理会话 + 命令行调用

**设计**：
```
用户 → 网页 → 服务器 → openclaw sessions send → 王鲸子代理 → 回复 → 网页
```

**问题**：
```bash
error: unknown option '--session'
```

**原因**：`openclaw sessions` 命令没有 `send` 子命令，只有 `list` 和 `cleanup`。

---

### 方案 2：子代理会话 + agent 命令

**设计**：
```bash
openclaw agent --session-id <session-key> --message <text>
```

**问题**：
```
Error: Invalid session ID: agent:main:subagent:9c7c2de4-4ac0-41fc-a67b-cc36738c80d3
```

**原因**：
- 子代理会话 ID 格式不被 `openclaw agent` 接受
- 子代理会话需要 `thread=true`，但 webchat 渠道不支持 `subagent_spawning hooks`

---

### 方案 3：本地嵌入模式

**设计**：
```bash
openclaw agent --local --message <text>
```

**问题**：
```
Error: Pass --to <E.164>, --session-id, or --agent to choose a session
```

**原因**：
- `--local` 模式需要配置模型 provider API key
- 当前环境使用阿里云百炼模型（bailian/qwen3.5-plus），不是本地模型
- 无法在没有 API key 的情况下运行嵌入模式

---

### 方案 4：文件队列 + 监控脚本

**设计**：
```
用户 → 网页 → input.txt → 监控脚本 → OpenClaw → output.txt → 网页轮询
```

**问题**：
- 监控脚本无法调用有效的 OpenClaw 命令
- 所有 `openclaw agent` 变体都需要有效的 session 或渠道配置

---

### 方案 5：浏览器字体渲染

**问题**：
- 用户报告浏览器显示"方框"（□□□）
- 复制粘贴到其他地方显示正常

**原因**：
- OpenClaw 内置浏览器运行在 WSL2 环境中
- WSL2 的 Chrome 缺少中文字体支持
- 尝试安装字体需要 `sudo` 权限

**尝试的修复**：
1. 在 HTML 中添加 Google Fonts 中文字体（Noto Sans SC）
2. 建议用户使用 Windows 本机 Chrome

---

## 🔍 根本原因分析

### 问题 1：OpenClaw 命令限制

| 命令 | 预期功能 | 实际支持 |
|------|----------|----------|
| `openclaw sessions send` | 发送消息到指定会话 | ❌ 不支持 |
| `openclaw agent --session-id` | 指定会话 ID | ❌ 仅支持主会话 |
| `openclaw agent --local` | 本地嵌入模式 | ❌ 需要 API key |

**结论**：OpenClaw 没有提供从外部脚本调用 Agent 的简单 CLI 接口。

---

### 问题 2：子代理会话限制

```python
# 尝试创建子代理
sessions_spawn(runtime="subagent", thread=true)
# 错误：thread=true is unavailable because no channel plugin registered subagent_spawning hooks
```

**原因**：
- webchat 渠道没有注册 `subagent_spawning hooks`
- 子代理会话需要渠道插件支持才能创建 thread-bound 会话
- 当前只有 Discord 等渠道支持此功能

---

### 问题 3：WSL2 环境限制

| 限制 | 影响 |
|------|------|
| 缺少中文字体 | 内置浏览器显示方框 |
| 无法使用 sudo | 无法安装字体 |
| 网络隔离 | Google Fonts 可能无法访问 |
| 浏览器超时 | 内置浏览器偶尔超时错误 |

---

### 问题 4：架构设计问题

**初始设计**：
```
用户 → 网页 → Python 服务器 → OpenClaw CLI → Agent → 回复 → 网页
```

**问题**：
- Python 服务器和 OpenClaw 是两个独立的进程
- 没有标准的 IPC（进程间通信）机制
- OpenClaw CLI 设计为交互式工具，不是 API 服务

---

## ✅ 可行的替代方案

### 方案 A：主会话扮演（当前采用）⭐

## 🎯 结论

### 为什么"完全自动化"行不通？

1. **OpenClaw CLI 设计限制**
   - 没有提供外部脚本调用的 API
   - `openclaw agent` 需要有效的会话或渠道配置
   - `openclaw sessions` 没有 send 命令

2. **渠道限制**
   - webchat 渠道不支持子代理 thread-bound 会话
   - 无法创建独立的王鲸会话

3. **环境限制**
   - WSL2 缺少中文字体
   - 无法使用 sudo 安装

### 当前最佳方案

**主会话扮演模式**：
1. 网页服务器正常运行（端口 8767）
2. 用户发送消息写入文件
3. 在主会话查看消息
4. 用王鲸身份回复
5. 回复显示在网页

**或者更简单**：直接在这个会话中对话，我扮演王鲸！

## 📝 经验教训

1. **先验证 CLI 能力**：在设计方案前，先用 `--help` 确认命令支持
2. **渠道限制要注意**：webchat 渠道功能有限，Discord/Telegram 更强大
3. **WSL2 环境限制**：字体、权限、网络都需要考虑
4. **简单优于复杂**：人工转发比复杂的自动化更可靠
