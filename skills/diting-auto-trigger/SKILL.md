# 谛听系统自动触发 Skill（结构化版 v3）

> 2026-04-09 重构：从"文本描述"改为"可执行规则"
> 2026-04-09 v3：融入质量提升方案 QT-01/QP-02/QP-01/QI-01/QP-04
> 触发时机: 任务完成后自动执行

## 🔴 最高优先级原则

1. **禁止空话**：每条规则必须有明确的验证方式，禁止"记录到 Skill"、"注意"、"小心"等不可验证表述
2. **数据驱动**：所有判定必须基于客观数据（工具调用记录、时间线、用户反馈），不依赖 AI 自评
3. **执行即检查**：每次任务完成后必须执行本检查清单，不能跳过

## 触发条件（满足任一即触发）

| # | 条件 | 判定标准 |
|---|------|---------|
| 1 | 数据处理任务 | feishu_sheet/read/exec 处理数据 ≥3 次调用 |
| 2 | 飞书文档/表格批量更新 | update_doc/write/append ≥2 次调用 |
| 3 | 长时间后台任务 | exec 后台命令且总执行时间 >60s |
| 4 | 用户反馈问题后的修复 | 用户指出错误后的修复操作 |
| 5 | 诊断类任务 | 包含根因分析/5 Why/排查关键词 |
| 6 | 用户信号 | 消息中包含"为什么"/"怎么"/"卡住了"/催促词 |

## 自动检查流程

```
任务完成
    ↓
第零步: 任务前评估回顾（QP-01）
  - 本次任务是否在执行前做了数据量/原因数/步骤数评估？
  - 评估结果是否向用户回复了？
  - 验证方式：检查首次回复是否在首次工具调用之前发出
    ↓
第一步: 快速分类（yes/no）
  - 数据处理？→ 检查规则 QT-01
  - 诊断类？→ 检查规则 QT-02
  - 长任务？→ 检查规则 QT-03
  - 用户催促？→ 检查规则 QT-04
    ↓
第二步: 规则匹配（对照下方检查表）
  - QP-02: 分段汇报是否执行？
  - QI-01: 诊断类是否有独立校验？
  - QP-04: 是否与历史事故重复？
    ↓
第三步: 输出结果
  - 无违规 → "✅ 质量检查通过"
  - 有违规 → "⚠️ 发现 N 个问题" + 创建/更新 INC
```

## 结构化检查表

### QT-01: 数据量 vs 工具匹配（结构化检查清单）

**验证方式**：检查工具调用记录中的 feishu_sheet/exec 调用次数和数据量声明

```
检查逻辑:
  if (feishu_sheet 调用次数 > 10):
    → 🚨 违规: 数据量大时应使用脚本批量处理
    → 验证: 统计本次会话 feishu_sheet/read/write/append 调用总次数
  if (exec 脚本处理数据):
    → ✅ 通过: 使用了正确的批量处理方式
  if (用户提供了数据量信息):
    → 检查是否按量级选择了正确工具
      <50行: API 可接受
      50-200行: 优先脚本
      >200行: 必须脚本
  if (数据量 > 100 且未声明处理方式):
    → 🚨 违规: 未执行 QP-01 任务前评估

违规处理:
  1. 创建/更新 INC 记录
  2. 更新 improvement-tracker.md 对应措施的验证结果为"未通过"
  3. 禁止使用"已记录到 Skill"作为完成标志
```

### QT-02: 诊断证据充分性

```
检查逻辑:
  if (诊断结论只有 1 个原因且未列出其他可能):
    → 🚨 违规: 必须列出 ≥2 个可能原因
  if (存在反证数据但未在结论中处理):
    → 🚨 违规: 确认偏误（如 0.2ms 查询时间证明不是数据库问题）
  if (诊断结论有 ≥2 个独立证据支持):
    → ✅ 通过

违规处理: 创建 INC（诊断错误类），关联 INC-006
```

### QP-02: 分段汇报强制规则（强制执行）

**验证方式**：检查消息时间线——用户消息与 AI 首次回复的时间差、连续工具调用中间是否有回复消息

```
检查逻辑:
  if (连续工具调用 > 3 且中间无回复消息):
    → 🚨 违规: 缺少分段汇报
    → 验证: 统计工具调用序列，检查中间是否有 message.send/回复
  if (用户发消息到首次回复时间 > 30s):
    → 🚨 违规: 首次回复延迟
    → 验证: 计算用户消息时间戳 - AI 首次回复时间戳
  if (任务执行时间 > 60s 且无中间汇报):
    → 🚨 违规: 超时强制汇报未执行
    → 验证: 计算 exec 开始到结束时间，检查中间是否有进度输出
  if (任务执行时间 > 120s):
    → 🚨 违规: 未交付已有结果
  if (长任务有中间进度更新):
    → ✅ 通过

分段汇报标准:
  - 简单任务(≤3步): 开始前回复 + 完成后汇报
  - 中等任务(4-8步): 每3步或每60秒汇报一次
  - 复杂任务(>8步): 每2步或每30秒汇报一次

违规处理:
  1. 创建 INC（体验问题类），关联 INC-001/002/003
  2. 记录具体时间差数据（如"等待 87 秒无汇报"）
  3. 更新 improvement-tracker.md 验证结果
```

### QT-04: 用户信号响应

```
检查逻辑:
  if (用户消息包含催促/质疑词 且 AI 未在下一条回复中响应):
    → 🚨 违规: 用户信号未响应
  if (用户纠正后 AI 只修表面不反思整体):
    → 🚨 违规: 修正不彻底
  if (用户催促后 AI 切换策略或交付已有结果):
    → ✅ 通过

违规处理: 创建 INC（响应问题类），关联对应历史事故
```

## 诊断类任务特别规则（INC-006 教训）

### 诊断流程（必须遵守）

```
问题现象 → 列出所有可能原因(≥2) → 逐一验证 → 根据证据修正 → 得出结论
```

### 禁止行为（🚨 自动检测）

| 禁止行为 | 检测方式 | 关联事故 |
|---------|---------|---------|
| 看到 A 和 B 同时存在 → 认为 A 导致 B | 检查是否有相关性≠因果性的说明 | INC-006 |
| 操作有效 → 认为原因判断正确 | 检查是否区分了"操作有效"和"原因正确" | INC-006 |
| 遇到反证 → 忽视并继续原判断 | 检查是否有与结论矛盾的数据未处理 | INC-006 |

### 必须行为（✅ 自动验证）

| 必须行为 | 验证方式 |
|---------|---------|
| 测量实际瓶颈，用数据排除/确认 | 诊断中必须包含具体测量数据 |
| 理解系统默认值的设计逻辑 | 不应建议修改合理的默认值 |
| 遇到反证 → 立即修正判断 | 反证数据必须在结论中体现 |
| 区分"操作有效"和"原因正确" | 明确标注两者的区别 |

## QP-01: 任务前评估清单（强制前置）

**验证方式**：检查首次回复是否在首次工具调用之前发出，回复中是否包含评估内容

```
执行时机: 收到任务后、首次工具调用前（≤50 tokens）

3 个 yes/no 快速评估:
  1. 数据处理？→ 数据量 >100 行？→ 是=必须用脚本
  2. 诊断类？→ 有多个可能原因？→ 是=必须逐一验证
  3. 长任务？→ 预计 >3 步？→ 是=必须先回复用户再执行

检查逻辑:
  if (首次工具调用在首次回复之前):
    → 🚨 违规: 未执行任务前评估
    → 验证: 比较首次回复时间戳 vs 首次工具调用时间戳
  if (数据量 >100 行且回复中未声明使用脚本):
    → 🚨 违规: QP-01 评估未执行
  if (诊断类任务回复中可能原因 <2 个):
    → 🚨 违规: QP-01 评估不完整

违规处理:
  1. 记录为 INC，关联 INC-007（指挥官模式违规）
  2. 必须在回复中体现评估结果，否则视为未执行
```

## QI-01: 独立校验机制（诊断类强制）

**验证方式**：检查诊断类复盘/结论中是否包含"被排除的假设"列表、是否有 ≥2 个独立证据

```
执行时机: 任何诊断类任务完成后

强制自检清单（3 个强制问题）:
  1. "如果这个判断是错的，什么证据能证明？"
  2. "有没有被我忽视的反证？"
  3. "换一个角度，这个现象还能怎么解释？"

检查逻辑:
  if (诊断结论只有 1 个原因且未列出其他可能):
    → 🚨 违规: 必须列出 ≥2 个可能原因
    → 验证: 统计诊断回复中列出的可能原因数量
  if (存在反证数据但未在结论中处理):
    → 🚨 违规: 确认偏误（如 0.2ms 查询时间证明不是数据库问题）
    → 验证: 检查诊断中是否有与结论矛盾的测量数据
  if (仅凭相关性得出因果性结论):
    → 🚨 违规: 相关性≠因果性
    → 验证: 检查是否有"A和B同时存在→A导致B"的逻辑
  if (诊断结论有 ≥2 个独立证据支持):
    → ✅ 通过
  if (复盘中有"被排除的假设"列表):
    → ✅ 通过

违规处理:
  1. 创建 INC（诊断错误类），关联 INC-006
  2. 必须列出被排除的假设及排除证据
  3. 反证数据必须在结论中明确处理
```

## QP-04: 重复事故检测（新建 INC 时强制）

**验证方式**：搜索历史 INC 记录的"根本原因"和"问题分类"字段，检查关键词和模式匹配

```
执行时机: 创建新 INC 记录时

强制检查:
  1. 关键词匹配: 搜索现有 INC 的"根本原因"和"问题分类"
  2. 模式匹配:
     - "不交付"模式: INC-001, INC-002, INC-003
     - "诊断错误"模式: INC-006
     - "规则违反"模式: INC-007
     - "信息错误"模式: INC-008
     - "重复回复"模式: INC-004
  3. 重复判定:
     - 同一根因重复 ≥2 次 → 标记"重复事故"
     - 改进措施已记录但仍发生 → 标记"改进失效"

检查逻辑:
  if (新 INC 根因与历史 INC 重复):
    → 标记: "重复事故"
    → 操作: 事故等级 +1（P2→P1, P1→P0）
    → 验证: 对比新旧 INC 的根因关键词
  if (改进措施已记录但问题仍发生):
    → 标记: "改进失效"
    → 操作: 重新分析措施为何无效
    → 验证: 检查 tracker 中对应措施的状态
  if (同一问题重复 ≥3 次):
    → 升级: 需要人类介入

输出要求:
  - 新 INC 必须包含"关联历史事故"字段（可为"无"）
  - 重复事故必须有升级记录
  - 改进失效必须有重新分析
```

## 改进措施追踪集成

每次检查完成后，必须更新 `projects/diting/improvement-tracker.md`:
- 相关措施 → 记录验证结果（通过/未通过），必须附具体数据
- 新发现的问题 → 创建新的追踪项，必须包含可操作的验证方式
- 过期措施 → 标记为 ⚠️已过期 并升级处理

**🔴 禁止行为**（违反即违规）:
- ❌ "已记录到 Skill" — 不是验证结果，是空话
- ❌ "已执行" — 无数据支撑的声明不算验证
- ❌ "注意 XXX" / "小心 XXX" — 不可验证的表述

**✅ 正确格式**:
- ✅ "2026-04-10 处理 200 行数据，AI 使用脚本 → 通过"
- ✅ "用户催促后首次回复延迟 3.2 秒 → 通过"
- ✅ "诊断任务列出 3 个原因并逐一验证 → 通过"
