# 金山文档实时监控 Skill

**创建时间**：2026-03-25  
**用途**：监控金山文档（录音转文字）更新，实时推送到飞书

---

## 🔧 核心能力

### 1. 浏览器工具用法

**能做什么：**
- ✅ 用 `browser open` 打开金山文档链接
- ✅ 用 `browser snapshot` 查看页面内容（ARIA 树）
- ✅ 读取录音转文字内容
- ✅ 检测新消息/新时间戳

**不能做什么：**
- ❌ 无法点击交互（登录、按钮等）
- ❌ 无法登录需要认证的页面
- ❌ Playwright 不可用

**关键认知：浏览器工具是只读的，但完全够用！**

## 📋 正确工作流程

### 方案 A：Cron + 手动检查（推荐）

```
1. 创建 cron 任务（每 30 秒触发）
2. cron 触发 systemEvent → 我收到提醒
3. 我手动调用 browser snapshot 检查文档
4. 对比上次内容，发现更新
5. 用 message 发送推送（优化格式）
```

**优点：**
- 真正实时监控
- 格式可控
- 100% 可用

**缺点：**
- 需要我每次手动调用 browser（不能自动化）

### 方案 B：文件队列（最可靠）

```
1. 外部 Python 脚本用 Selenium 检查金山文档
2. 发现新内容时写入 /tmp/doc-update.txt
3. cron 每 30 秒检查这个文件
4. 有内容时触发 systemEvent + 读取文件
5. 我发送推送
```

**优点：**
- 完全自动化
- 绕过浏览器工具限制
- 稳定可靠

**缺点：**
- 需要外部脚本

## 📱 消息格式规范（避免折叠）

### 飞书折叠阈值
- **视觉行数**：超过 6-8 行 → 折叠
- **空行数量**：大量空行 → 易折叠
- **字符数**：300+ 字 → 可能折叠

### 优化格式模板

```markdown
📊 **状态监控 | 2026-03-25 01:56**

**运行状态**：✅ 正常 | **响应**：<1s | **折叠**：无
**任务**：等待指令 | **消息**：格式优化完成

有需要随时叫我～ 👂
```

**关键技巧：**
1. ✅ 控制在 5 行视觉高度内
2. ✅ 用**加粗**代替换行强调
3. ✅ 减少空行分隔
4. ✅ 合并相关段落
5. ✅ 用 `|` 分隔状态信息

**错误示例（会折叠）：**
```markdown
✅ **状态正常**

---

**运行状态**：正常

**响应时间**：<1s

**折叠**：无

---

有需要随时叫我～
```

## 🛠️ 配置示例

### Cron 任务配置

```json
{
  "name": "金山文档监控",
  "schedule": {
    "kind": "every",
    "everyMs": 30000
  },
  "payload": {
    "kind": "systemEvent",
    "text": "📊 文档监控提醒 - 检查金山文档更新"
  },
  "sessionTarget": "main",
  "delivery": {
    "mode": "announce"
  }
}
```

### 浏览器检查流程

```
1. browser open (url="https://www.kdocs.cn/l/xxx")
   → 获取 targetId

2. browser snapshot (targetId="xxx", refs="aria")
   → 获取页面 ARIA 树

3. 解析 snapshot 内容
   → 提取最新时间戳/消息

4. 对比上次记录
   → 判断是否有更新

5. 有更新 → message.send() 推送
   → 无更新 → 跳过或发送简短状态
```

## ⚠️ 常见错误与解决方案

### 错误 1：说"看不了网页"

**原因**：忘记了 browser 工具的能力  
**解决**：记住 `browser open + snapshot` 可以查看任何公开网页

### 错误 2：消息被折叠

**原因**：太多空行、分隔线、视觉行数超标  
**解决**：用优化格式模板，控制在 5 行内

### 错误 3：机械重复推送

**原因**：cron 的 systemEvent 是固定文本，无法自动调用 browser  
**解决**：收到提醒后手动调用 browser 检查，或用文件队列方案

### 错误 4：金山文档需要登录

**原因**：文档是私有的  
**解决**：
1. 用户设为"公开链接（任何人可查看）"
2. 或用文件队列方案（外部脚本用浏览器登录）

## 🎯 最佳实践总结

1. **浏览器工具是只读的，但够用** - 能查看任何公开网页
2. **消息格式决定折叠** - 紧凑、少空行、5 行内
3. **cron 无法自动化 browser** - 需要手动调用或文件队列
4. **金山文档必须公开** - 否则无法访问
5. **30 秒轮询是技术上限** - 无法更快（避免 API 限流）
