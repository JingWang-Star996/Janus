# ZVP 项目组周报管理与验证制度

**版本**：v1.0  
**生效日期**：2026-03-20  
**适用范围**：ZVP 项目组全体成员  
**制定原因**：针对周报内容虚假、进度无法验证等问题

---

## 📋 第一部分：周报规范要求

### 1.1 禁止使用的模糊描述

| 模糊描述 | 问题 | 正确写法示例 |
|---------|------|------------|
| "跟进 XX 开发" | 无法验证具体做了什么 | "与唐浩晖确认战争迷雾接口，输出需求文档 v1.2" |
| "持续跟进" | 无具体产出 | "跟进 3 次，解决 2 个 blocking 问题（附 tapd 链接）" |
| "排期&开发跟进" | 范围太广 | "与陈华强确认创角界面排期，定于 3/25 提测" |
| "XX 设计完成" | 无交付物证明 | "XX 设计完成，文档链接：[飞书文档链接]" |
| "XX 开发 100%" | 无验收标准 | "XX 开发 100%，已提测（tapd 链接），测试负责人：XXX" |

### 1.2 必须包含的要素

**策划岗周报必须包含**：
- [ ] 设计文档链接（飞书文档）
- [ ] 关联开发负责人确认（@开发）
- [ ] 进度百分比需有里程碑依据

**程序岗周报必须包含**：
- [ ] 代码提交记录（git commit/tapd）
- [ ] 功能测试状态（自测/提测/验收）
- [ ] 关联策划确认（@策划）

**美术岗周报必须包含**：
- [ ] 产出物截图/链接
- [ ] 导入引擎状态（本地/已导入）
- [ ] 关联程序确认（@程序）

### 1.3 进度百分比规范

| 进度 | 定义 | 需要提供 |
|------|------|---------|
| 0% | 未开始 | - |
| 10%-30% | 方案设计中 | 设计文档初稿 |
| 40%-60% | 开发进行中 | 中期演示/代码提交记录 |
| 70%-90% | 开发收尾 | 提测记录/测试报告 |
| 100% | 已完成 | 验收报告/上线证明 |

**禁止行为**：
- ❌ 无交付物声称 100%
- ❌ 连续 2 周同一进度无变化
- ❌ 进度回退无说明

### 1.4 特殊模块标注规范

**涉及多岗位协作的功能**，必须标注：

```
【搜打撤关卡】
- 策划：王小星（设计 100%，文档链接）
- 程序：齐翔（开发 50%，tapd 链接）
- 状态：⚠️ 开发中（预计 3/25 提测）
```

**涉及外部依赖的功能**，必须标注：

```
【战争迷雾】
- 依赖：服务器数据上报接口
- 状态：✅ 接口已就绪 / ⚠️ 接口延期 / ❌ 接口未开始
- 负责人：XXX
```

## 🔗 第二部分：岗位关联验证机制

### 2.1 推荐方案：上下游确认制（非 OKR）

**为什么不是 OKR**：
- OKR 适合目标管理，不适合进度验证
- OKR 周期长（季度），周报需要即时验证
- OKR 不解决"虚假汇报"问题

**上下游确认制核心**：
- 每个功能的**上游**和**下游**负责人必须互相确认
- 周报中的进度需要**关联方签字确认**
- 关键节点需要**产出物证明**

### 2.2 岗位关联关系图

```
策划（需求） → 程序（开发） → 测试（验收）
    ↑              ↓              ↓
    └────── 美术（资源）←────────┘
```

### 2.3 验证流程

#### 阶段 1：需求阶段（策划主导）

| 步骤 | 动作 | 产出物 | 确认人 |
|------|------|--------|--------|
| 1 | 策划输出设计文档 | 飞书文档 | - |
| 2 | 程序评审可行性 | 评审记录 | 程序负责人 |
| 3 | 美术评估资源量 | 资源清单 | 美术负责人 |
| 4 | 排期确认 | 排期表 | 三方签字 |

**周报体现**：
```
【搜打撤关卡】
- 设计文档：[链接]
- 评审状态：✅ 已完成（3/10）
- 排期：3/11-3/25
- 程序确认：@齐翔 ✅
```

#### 阶段 2：开发阶段（程序主导）

| 步骤 | 动作 | 产出物 | 确认人 |
|------|------|--------|--------|
| 1 | 程序提交代码 | git commit/tapd | - |
| 2 | 策划验收功能 | 验收记录 | 策划负责人 |
| 3 | 美术验收资源 | 资源导入确认 | 美术负责人 |
| 4 | 测试提测 | 测试报告 | 测试负责人 |

**周报体现**：
```
【搜打撤关卡】
- 开发进度：50%
- 代码提交：[tapd 链接]
- 策划验收：⚠️ 进行中（3/18）
- 预计提测：3/25
```

#### 阶段 3：验收阶段（测试主导）

| 步骤 | 动作 | 产出物 | 确认人 |
|------|------|--------|--------|
| 1 | 测试出具报告 | 测试报告 | - |
| 2 | 策划确认功能 | 验收签字 | 策划负责人 |
| 3 | 程序修复 bug | bug 修复记录 | 测试负责人 |
| 4 | 上线发布 | 上线记录 | 项目经理 |

**周报体现**：
```
【搜打撤关卡】
- 测试状态：✅ 通过（3/24）
- bug 数：3 个（已修复）
- 策划验收：✅ 已确认（@王小星）
- 上线时间：3/26
```

### 2.4 周报确认流程

```
每周五 18:00 前：成员提交周报
     ↓
每周五 20:00 前：关联方确认（程序确认策划、策划确认程序等）
     ↓
每周五 22:00 前：项目经理审核
     ↓
每周六 10:00：发送周报汇总
```

**确认责任**：
- 关联方**必须在 2 小时内**确认或提出异议
- 超时未确认视为**默认同意**
- 虚假确认**连带责任**

## 📊 第三部分：异常监控机制

### 3.1 自动预警规则

| 规则 | 触发条件 | 处理方式 |
|------|---------|---------|
| 进度异常 | 连续 2 周同一进度无变化 | 项目经理约谈 |
| 完成率异常 | 100% 完成率>50%（团队平均 30%） | 要求提供产出物证明 |
| 确认异常 | 关联方 3 次未确认 | 计入绩效考核 |
| 描述异常 | 使用模糊描述>3 次/周 | 打回重写 |

### 3.2 红黄绿灯制度

| 灯号 | 定义 | 处理 |
|------|------|------|
| 🟢 绿灯 | 进度正常，无异常 | 正常流程 |
| 🟡 黄灯 | 触发 1 条预警规则 | 项目经理提醒 |
| 🔴 红灯 | 触发 2 条以上或虚假汇报 | 绩效扣分 + 全员通报 |

### 3.3 虚假汇报处理

| 次数 | 处理 |
|------|------|
| 第 1 次 | 警告 + 要求补交产出物 |
| 第 2 次 | 绩效扣分（5%）+ 全员通报 |
| 第 3 次 | 绩效扣分（20%）+ 调岗评估 |

## 🛠️ 第四部分：实施工具

### 4.1 飞书多维表格模板

**周报提交表**：
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 姓名 | 人员 | ✅ | - |
| 岗位 | 单选 | ✅ | 策划/程序/美术/测试 |
| 本周工作 | 多行文本 | ✅ | 禁止模糊描述 |
| 进度% | 数字 | ✅ | 需提供依据 |
| 产出物链接 | 链接 | ✅ | 文档/tapd/git |
| 关联方确认 | 人员 | ✅ | @上下游负责人 |
| 确认状态 | 单选 | ✅ | 待确认/已确认/有异议 |

### 4.2 自动化流程

```
成员提交周报
     ↓
飞书机器人自动@关联方
     ↓
关联方点击确认/提出异议
     ↓
超时自动升级项目经理
     ↓
周六自动生成汇总报告
```

### 4.3 周报模板示例

```markdown
## 王小星 - 策划 - 2026/3/18

### 本周工作

1. 【搜打撤关卡】设计落地
   - 进度：100%
   - 产出物：[设计文档链接]
   - 关联方：@齐翔（程序）
   - 确认状态：✅ 已确认

2. 【联盟基础设计】
   - 进度：100%
   - 产出物：[设计文档链接]
   - 关联方：@陈华强（程序）
   - 确认状态：✅ 已确认

### 需要协助

- 搜打撤开发排期待确认（@齐翔）
```
